# The Ultimate Guide to Fixing Email Issues After Migrating from cPanel to HestiaCP

Migrating a server from cPanel to HestiaCP is usually a smooth process, thanks to Hestia’s built-in `v-import-cpanel` script. However, when it comes to migrating legacy email accounts, the import script has a few notorious quirks that can leave your mail server trapped in routing loops, throwing SSL errors, or hiding your emails entirely.

If you’ve recently migrated and your emails are bouncing, delayed, or disappearing, the problem almost always boils down to three specific areas: **Exim routing maps, file ownership permissions, or DNS proxy conflicts**.

Here is the step-by-step guide to diagnosing and fixing the most common cPanel-to-HestiaCP email migration failures.

---

### Phase 1: Fixing the "Unrouteable Address" Error

**The Symptom:** External senders (like Gmail) immediately receive a bounce-back stating `550 Verification failed for Unrouteable address` or `Sender verify failed`.

**The Cause:** HestiaCP uses Exim as its mail transfer agent. Exim relies on a specific local directory map (`/etc/exim4/domains/`) to know where to route incoming mail. During a cPanel import, the script occasionally fails to generate the `passwd` and `aliases` files inside this directory. Exim receives the email, looks for the local map, finds nothing, and bounces the message because it assumes the mailbox doesn't exist on the server.

**The Fix:** You need to force HestiaCP to rebuild the Exim routing files for the affected domain.

1. SSH into your server as root.
2. Run the built-in rebuild command (replace `admin` with your Hestia username and `example.com` with your domain):
   ```bash
   v-rebuild-mail-domain admin example.com
   ```
3. Verify the files were generated by checking the directory:
   ```bash
   cat /etc/exim4/domains/example.com/passwd
   ```
   _If you see your email addresses listed alongside encrypted passwords, the routing loop is fixed._

---

### Phase 2: Emails Arrive, but Webmail is Completely Empty

**The Symptom:** The Exim logs (`/var/log/exim4/mainlog`) show that emails are successfully being delivered to the `localuser`, but when you log into Roundcube or SnappyMail, your inbox is completely blank. Your historical cPanel emails are also missing.

**The Cause:** This is a classic file permission error caused by the `v-import-cpanel` script. The script extracts the old emails onto the hard drive but often leaves the top-level mail directory owned by the `root` user. Dovecot (the storage agent) and the webmail interface do not have the security clearance to read files owned by `root`, so they assume the inbox is empty. Additionally, leftover cPanel index files can corrupt Dovecot's cache.

**The Fix:** Reassign ownership to the correct user and flush the corrupted indexes.

1. Force the correct ownership recursively for the entire mail directory (replace `USERNAME` with your Hestia user):
   ```bash
   chown -R USERNAME:mail /home/USERNAME/mail/
   ```
2. Delete the corrupted, hidden Dovecot index files. (This is perfectly safe and does not delete your actual emails; Dovecot will instantly rebuild them):
   ```bash
   rm -f /home/USERNAME/mail/example.com/*/dovecot*
   ```
3. Log out of webmail, ensure you log back in using the **full email address** as the username, and your historical and new emails will populate.

### Phase 3: Gmail Refuses to Deliver (TLS/SNI Handshake Errors)

**The Symptom:** You send a test email from Gmail, but it never arrives. When checking the Exim log, you see this specific error occurring at the exact moment you sent the test:
`TLS error on connection from mail-xxx.google.com (gnutls_handshake): A disallowed SNI server name has been received.`

**The Cause:** Google enforces strict SSL/TLS encryption. When Gmail tries to hand the email to your server, it demands a valid SSL certificate. This fails for two reasons:

1. Your Hestia mail domain doesn't have Let's Encrypt enabled.
2. Your DNS `MX` record is pointed to your root domain (e.g., `example.com`) which is currently being proxied (the "Orange Cloud") by Cloudflare. Cloudflare proxies strip standard mail ports, breaking the SSL handshake.

**The Fix:** Align your DNS and generate a proper mail certificate.

1. Log into your Cloudflare dashboard and go to your DNS records.
2. Ensure you have an `A` record for `mail.example.com` pointing to your server IP, and set the Proxy status to **DNS Only (Grey Cloud)**.
3. Edit your `MX` record. The "Mail server" value **must** point to `mail.example.com` (never the root domain).
4. Log into your HestiaCP dashboard, go to **Mail**, edit the domain, and check **Enable SSL for this domain** and **Use Let's Encrypt**. Save the changes.

---

### Phase 4: The Ultimate Preventative Fix

If you regularly migrate clients from cPanel, dealing with the Phase 2 permission issue gets tedious. You can permanently patch Hestia's import script to automatically assign the correct ownership at the end of the migration.

1. Open the import script:
   ```bash
   nano /usr/local/hestia/bin/v-import-cpanel
   ```
2. Scroll down to the very end of the `restore_mail()` function.
3. Inject this global ownership fix right before the closing brace `}`:
   ```bash
   # Fix global mail directory ownership to prevent webmail empty inbox errors
   chown -R $NEW_USER:mail /home/$NEW_USER/mail/
   ```
4. Save the file. Future migrations will now automatically display emails in Roundcube without manual intervention.
